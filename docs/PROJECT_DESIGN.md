# 项目设计说明与代码审核

本文基于当前代码整理，不把项目中不存在的能力写成已经实现的能力。当前工程是 C++20 的 60Hz 顶视角回滚同步 Demo，核心目标是验证联机动作游戏的固定帧推进、服务端权威、本地预测、回滚重放、UDP 输入冗余和状态哈希对账。

## 代码审核结论

已验证：

- `cmake --build build` 构建通过。
- `ctest --test-dir build --output-on-failure` 通过，包含 `lab_tests` 和 `lab_stress_smoke`。
- 当前代码没有接入 Redis，也没有实现典型 Singleton 对象。项目采用显式上下文和显式所有权，例如 `ServerCtx`、`ClientCtx`、`World`、`UdpSocket`。

主要风险和改进建议：

1. 迷宫只同步 `mazeSeed`，没有同步完整迷宫网格。`World::GenerateMaze` 使用 `std::shuffle` 和 `std::mt19937` 生成迷宫，而 `Hasher` 又会把迷宫格子纳入哈希。当前同机测试没有问题，但跨编译器、跨标准库或后续换平台时，随机分布和洗牌实现差异可能导致同 seed 生成不同迷宫，从而出现哈希不一致。生产级做法是同步完整迷宫数据，或实现项目自有的确定性随机/洗牌算法。
2. `kStartDelayTicks` 当前更像 tick 编号偏移，不是真正的开局倒计时。服务端在 `MaybeStartMatch` 中把 `tick` 直接跳到 `startTick` 并把 `started` 设为 true。如果设计目标是给客户端预热 30 帧，应增加等待态，让服务端在墙钟时间经过对应帧数后再正式推进。
3. 服务端对客户端输入包的校验偏弱。`OnUdp` 会把包里的 `cmd.tick` 直接写入环形缓冲，恶意或异常客户端可以发送过大 tick，覆盖当前环槽。建议限制输入 tick 范围，例如只接受 `[serverTick - historyWindow, serverTick + maxLead]`，同时校验 `moveX/moveY` 只能为 `-1/0/1`、`buttons` 只包含允许位。
4. 网络解码函数读完有效字段后没有强制检查 `p == end`，带尾部垃圾的数据包也会被接受。当前 Demo 可运行，但协议更严格时应拒绝尾部多余字节，便于发现粘包、版本不匹配或攻击流量。
5. 玩家推箱碰撞在墙体碰撞之后执行，`ResolvePushbox` 可能把玩家推入墙体。建议推开玩家后再做一次墙体约束，或者把玩家间碰撞也纳入统一碰撞求解。
6. `Action::Attack`、`atkActive`、`attackConnected` 等字段在协议和快照中保留了，但当前攻击逻辑实际是发射子弹，未进入 `Action::Attack` 状态。如果这是预留字段，应在说明中标注；如果希望表现攻击前摇/后摇，需要补齐状态机。

## 项目定位

本项目不是传统的 Web 后端，也不是完整商业游戏。它更适合作为网络同步架构 Demo：

- 服务端负责权威世界推进，防止客户端各自计算导致分叉。
- 客户端负责本地预测，保证玩家按键后立即有反馈。
- 当服务端权威状态到达时，客户端恢复权威快照并重放本地输入，修正预测误差。
- 每个 tick 生成稳定哈希，用于发现客户端和服务端世界是否分叉。

这种结构适用于动作游戏、格斗游戏、俯视角射击、轻量竞技房间等对实时反馈和一致性都有要求的场景。

## 模块划分

项目按职责分成三层：

- 模拟层：`include/lab/sim`、`src/sim`
  - `World`：世界推进核心，处理移动、墙体碰撞、玩家推箱、子弹生成、命中和快照恢复。
  - `WorldSnapshot`：网络同步、回滚、哈希共用的数据快照。
  - `InputBuffer`、`InputHistory`：按 tick 存储输入的环形缓冲。
  - `StateHistory`：按 tick 存储预测或权威快照，供回滚和哈希对账使用。
  - `Hasher`：把世界快照量化后做稳定哈希。

- 网络层：`include/lab/net`、`src/net`
  - `UdpSocket`：非阻塞 UDP 封装，并接入 libevent。
  - `Packets.h`：协议包结构，当前版本是 v3。
  - `NetCodec`：手写二进制编解码，统一使用网络字节序。

- 应用层：`apps`、`include/lab/app`、`src/app`
  - `server_main.cpp`：服务端权威帧循环、握手、缺输入处理、状态广播。
  - `client_main.cpp`：输入采样、本地预测、收包、回滚、SDL 渲染。
  - `GameConfig`：玩家数、帧长、输入冗余等配置。
  - `InputPrediction`：根据权威状态粗略推断远端玩家输入。
  - `ClientRender`：SDL2 俯视角调试渲染。

## 核心数据流

客户端启动后，在没有收到 `Start` 前周期发送空 `Input` 包作为 hello。服务端按 UDP 地址分配玩家槽位，收齐 `kRequiredPlayers` 后广播 `Start`，告诉每个客户端自己的 `playerId` 和统一的 `startTick`。

开局后，每个客户端每 tick 做四件事：

1. 从 SDL 键盘状态采样本地输入。
2. 写入本地 `InputBuffer`，立即用本地输入和远端预测输入推进 `worldPred`。
3. 保存预测快照到 `StateHistory`。
4. 把最近 K 帧本地输入打成冗余 `InputPacket` 发送给服务端。

服务端每 tick 做四件事：

1. 从每个玩家的输入缓冲里取当前 tick 的输入。
2. 如果缺输入，在短窗口内复用上一帧输入；超过窗口则使用默认空输入。
3. 调用 `World::Step` 推进权威世界。
4. 每帧发 `Ack`，每 `kStateEvery` 帧发完整 `State`。

客户端收到 `State` 后会构造权威 `WorldSnapshot`，校验 `stateHash`，然后从权威 tick 开始重放到当前本地 tick。即使本地玩家误差没有超过回滚阈值，也会执行 rebase + replay，这样远端玩家和子弹能尽快采用权威状态。

## 为什么使用固定时间步

游戏逻辑以 60Hz 固定 tick 推进，原因是联机同步依赖确定性。若直接用每帧真实耗时 `deltaTime` 推进，不同机器的帧率、调度抖动、窗口拖动都会改变物理积分结果，服务端和客户端更容易分叉。

固定时间步的优势：

- 同一 tick 的同一组输入会得到相同输出。
- 快照、输入、哈希都可以用 tick 作为统一索引。
- 回滚时只需要从某个权威 tick 重放离散帧。
- 网络状态包只需要描述 tick，不依赖真实时间戳。

代码中服务端和客户端都使用 accumulator：真实时间只负责累计，逻辑每次只按 `1/60` 秒推进。

## 为什么服务端权威

客户端输入不可信，客户端本地预测也可能因为网络延迟、丢包、远端预测错误而偏离真实世界。因此服务端必须维护唯一权威世界：

- 客户端只上传输入，不上传最终位置和血量。
- 服务端统一执行碰撞、子弹命中、HP 扣减。
- 客户端收到权威状态后必须接受服务端结果。

这样可以避免玩家通过改内存直接改位置、血量或命中结果。当前 Demo 仍缺少完整安全校验，但架构方向是服务端权威。

## 为什么使用 UDP 和输入冗余

实时动作同步不能把每个输入都阻塞在可靠传输上。TCP 虽然可靠，但丢包时会触发队头阻塞，后续新输入必须等待旧包重传，手感会变差。

本项目使用 UDP，配合输入冗余降低丢包影响：

- 每个 `InputPacket` 不只带当前 tick 输入，还带最近 K 帧输入。
- 如果某个 UDP 包丢了，后续包仍可能补上丢失 tick 的输入。
- 服务端按 tick 从 `InputBuffer` 取输入，缺失时短暂 HoldLast，再退化为空输入。

这种设计接受少量丢包，但避免因为可靠重传阻塞最新输入。

## 为什么使用环形缓冲

输入历史和状态历史都按 tick 存取，但回滚只需要最近一段窗口，不需要无限保存。因此使用固定容量环形缓冲：

- 写入成本 O(1)，索引是 `tick % capacity`。
- 内存有上限，适合长时间运行。
- 读取时校验槽位里保存的 tick，避免环槽被覆盖后误读旧数据。

这也是 `InputBuffer::Get` 和 `StateHistory::Get` 返回 `optional` 的原因：某个 tick 可能未收到，或者已经被覆盖。

## 为什么需要本地预测和回滚

如果客户端等待服务端确认后再移动，本地玩家每次按键都要等待一个 RTT，操作会明显滞后。本地预测的做法是：客户端先假设输入会被服务端接受，立即推进本地世界。

预测一定可能出错，主要来源是：

- 服务端收到输入的时机和客户端预期不同。
- 远端玩家真实输入未知，只能根据状态猜测。
- UDP 丢包导致服务端短暂 HoldLast 或使用默认输入。

回滚用于修正这些误差。收到权威 `State` 后：

1. 恢复权威快照。
2. 从 `authoritativeTick + 1` 开始重放本地保存的输入。
3. 重放到当前本地 tick。
4. 覆盖新的预测状态。

当前回滚计数只比较本地玩家的位置、HP、动作和落地状态，不把远端玩家纳入触发条件。原因是远端本来就是预测的，如果用远端误差触发回滚，客户端会频繁抖动。

## 为什么使用状态哈希

回滚系统很容易出现“看起来能跑，但某些 tick 已经分叉”的问题。状态哈希用于快速定位一致性问题：

- 服务端对权威 `WorldSnapshot` 计算 `stateHash`。
- `StatePacket` 中携带该哈希。
- 客户端把收到的状态包还原为快照后重新计算哈希。
- `AckPacket` 也携带服务端当前哈希，便于调试对账。

`Hasher` 对 float 使用毫米级量化，避免服务端原始 float 和客户端网络解码后的 float 因极小误差产生假 mismatch。它还把玩家、子弹、迷宫 seed、迷宫尺寸和迷宫格子一起纳入哈希。

## 为什么当前不使用单例模式

当前项目没有典型 Singleton，这是合理选择。

回滚同步项目最怕隐藏的全局可变状态。`World`、`InputBuffer`、`StateHistory`、`UdpSocket` 都需要在测试、压力模拟、客户端预测和服务端权威之间创建多个实例。如果把这些对象做成单例，会带来几个问题：

- 无法同时在一个进程里创建权威世界和客户端预测世界，压力测试会很难写。
- 回滚重放要求从快照恢复到旧状态，单例的隐藏状态容易漏恢复。
- 测试之间容易互相污染，确定性验证不可靠。
- 生命周期不清晰，尤其是 socket、libevent event 和 SDL 资源。

本项目选择显式上下文：

- 服务端所有运行态放在 `ServerCtx`。
- 客户端所有运行态放在 `ClientCtx`。
- `World` 是普通对象，可以构造、恢复、替换。
- `UdpSocket` 使用 RAII 管理 fd 和事件注册。

如果答辩时被问“为什么不用单例”，可以回答：本项目核心目标是确定性、可回滚、可测试，多实例是基本需求；单例会隐藏依赖和状态，破坏回放验证。配置类 `GameConfig` 目前只是编译期常量，不需要做成带状态的单例。

如果未来真的要引入单例，适合的对象也应该很少，例如只读配置表或日志入口；不应把 `World`、网络连接、玩家会话、输入历史做成单例。

## Redis 与读取一致性考虑

当前项目没有 Redis。实时对局内的每帧世界状态也不适合放到 Redis 做读缓存，因为 60Hz 状态更新太频繁，权威状态应保留在服务端进程内，由服务端按 tick 推送给客户端。

如果项目扩展成线上服务，Redis 更适合用于对局外或低频数据：

- 登录态、短期 token、玩家在线状态。
- 匹配队列、房间列表、服务器注册表。
- 排行榜、战绩摘要、回放索引。
- 限流计数、临时冷却、邀请关系。

Redis 不能自动保证所有读取都是强一致。不同场景要选择不同策略：

- 需要读到刚写入的数据：读主库，或者写入后用版本号确认，不从可能延迟的从库读。
- 主从复制场景：如果必须提高写入落盘或复制把握，可以在关键写后使用 `WAIT` 等待副本确认，但这会增加延迟。
- 缓存数据库数据：使用 cache-aside。写数据库成功后删除缓存，读 miss 再加载；配合 TTL 防止脏数据长期存在。
- 防止旧值覆盖新值：给数据加版本号、更新时间或 tick，写入时只接受更新版本。
- 排行榜等 Redis 原生结构：Redis 可以作为该数据的主存储，但需要定期落库或做持久化备份。
- 对局状态：不要用 Redis 做每帧权威状态来源。服务端内存是权威，Redis 只可记录房间元数据或最终结果。

如果答辩时要讲“Redis 如何保证读取”，建议不要说 Redis 天然强一致，而是说：根据业务一致性要求选择读主、版本号、缓存失效、TTL、`WAIT`、幂等写入和最终落库。实时对战内的强一致由权威服务器和 tick 状态哈希保证，不交给 Redis。

## 协议设计

协议头包含：

- `magic`：识别本项目协议。
- `version`：当前为 v3，便于后续升级。
- `type`：区分 `Input`、`Ack`、`State`、`Start`。

主要报文：

- `Input`：客户端到服务端。包含玩家 id、自增序号、最新输入 tick、客户端已确认的服务端 tick，以及最近 K 帧输入。
- `Start`：服务端到客户端。分配 `playerId`，下发 `totalPlayers` 和 `startTick`。
- `Ack`：服务端到客户端。反馈服务端已处理 tick、该玩家服务端已收到的最大输入 tick、服务端状态哈希。
- `State`：服务端到客户端。包含权威 tick、迷宫 seed、玩家数组、子弹数组和状态哈希。

所有整数使用网络字节序。位置和速度以毫米整数传输，避免直接传 float 带来的平台差异和序列化不稳定。

## 世界模拟设计

世界是一个简化的顶视角坦克/迷宫对战模型：

- 地图是固定尺寸迷宫，墙体用于玩家和子弹碰撞。
- 玩家有位置、速度、朝向、HP、硬直、射击冷却、瞄准方向。
- 输入只包含移动方向和按钮位。
- 移动使用目标速度 + 摩擦插值。
- 子弹直线移动，碰墙或命中玩家后消失。
- 命中后扣 HP，并让目标进入短暂 Hitstun。

`World::Step` 是唯一推进入口。只要输入命令数量等于玩家数且 tick 对齐，就能推进一帧。这个设计让服务端、客户端预测、压力测试、回滚重放都复用同一套模拟逻辑。

## 测试设计

当前测试分两类：

- `lab_tests`：覆盖基础结构，例如输入环形缓冲、状态历史、输入包编解码、状态包编解码、哈希量化。
- `lab_stress_smoke`：在单进程里模拟权威端和客户端，覆盖输入冗余、状态编解码、延迟 State、Restore + Replay、哈希一致性。

压力测试的价值很高，因为它不依赖窗口、真实 socket 和手工操作，可以快速验证回滚核心链路是否稳定。

## 后续演进建议

短期可以优先做这些改进：

1. 给解码函数增加尾部字节校验，并补对应单元测试。
2. 服务端限制输入 tick 和输入值范围，避免异常包覆盖环形缓冲。
3. 明确 `kStartDelayTicks` 语义。如果要真实倒计时，就实现等待态。
4. 迷宫同步改成完整网格，或替换为自研确定性 RNG 和 shuffle。
5. 推箱碰撞后增加墙体约束，避免玩家被挤进墙。
6. 将 `ServerCtx` 中重复的配置改为引用 `GameConfig`，减少服务端和客户端配置漂移。
7. 如果继续完善攻击系统，补齐 `Action::Attack` 的前摇、活动帧、后摇和网络同步语义；如果不需要，就清理保留字段或标注为预留。

中长期可以考虑：

- 房间管理和观战。
- 输入延迟自适应和客户端追帧。
- 断线重连和状态快照补发。
- 更严格的反作弊输入校验。
- 真实网络压测和 bot client。
- 协议 fuzz 测试。
- Redis 只用于匹配、房间元数据、排行榜和回放索引，不承载每帧权威状态。
